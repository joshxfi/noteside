# AGENTS.md

Guidance for any coding agent working in this repository — the single, tool-agnostic source of truth for how Noteside is built. Keep it accurate as the architecture changes.

## What this is

Noteside — an offline & **keyboard-first** notes desktop app (full vim _and_ conventional shortcuts, both first-class), plus its marketing site. Turborepo + pnpm monorepo:

- `apps/desktop` — Tauri 2 + React 19 + Vite + TypeScript (the app)
- `apps/landing` — Vite + React 19 + TypeScript + Tailwind v4 (the site)
- `apps/docs` — Fumadocs on React Router 7 (SPA, prerendered) — the documentation site at **docs.noteside.app** (`:3002` via `pnpm dev:docs`)
- `apps/brand` — Vite + React + Tailwind, the standalone **brand guide** (internal reference only; not deployed, not linked from the landing). Runs on `:3001` via `pnpm dev:brand`; has its own `logo.tsx` copy + `styles.css` (the brand CSS).

## Commands

Run from the repo root (Turborepo fans out to the right package):

```bash
pnpm install
pnpm dev              # everything: landing (:3000) + desktop Tauri window
pnpm dev:landing      # landing only
pnpm dev:desktop      # desktop only — runs `tauri dev` (needs Rust + Tauri system deps)
pnpm typecheck        # tsc --noEmit across the workspace  ← primary verification gate
pnpm build            # web bundles for all apps (Vite; docs via react-router)
pnpm lint             # oxlint (root; config .oxlintrc.json)   — pnpm lint:fix to autofix
pnpm format           # oxfmt (writes in place; root; config .oxfmtrc.json) — pnpm format:check to verify
```

Desktop / Tauri specifics:

```bash
pnpm --filter @noteside/desktop dev:web     # Vite only, no native window (browser dev)
pnpm --filter @noteside/desktop build       # web bundle → apps/desktop/dist
pnpm tauri build                            # full native bundle (see Icons below)
cd apps/desktop/src-tauri && cargo check    # verify the Rust shell compiles
# icon source of truth = scripts/mark.html (brand mark in Newsreader, exact design CSS).
# Render it with headless Chrome (see the file header) → src-tauri/app-icon.png, then:
pnpm --filter @noteside/desktop tauri icon src-tauri/app-icon.png  # → all bundle icons (.icns/.ico/png)
pnpm demo:build                             # build desktop web bundle → apps/landing/public/demo
pnpm --filter @noteside/desktop bench       # vitest benchmarks — JS hot paths (computeBacklinks, parse)
cd apps/desktop/src-tauri && cargo bench    # criterion — search/scan at 1k/10k/50k (benches/perf.rs)
```

**Performance posture (measured — see `benches/perf.rs` + `src/perf.bench.ts`).** `fuzzy_files` is ~1ms even at 50k (nucleo; a score-only pass picks the top 200, then `indices()` runs only on the returned page — the match-heavy case is ~3.5ms @50k, 1.7× faster than single-pass; the no-match case pays a ~25% score-pass tax and stays ~1ms). `content_search` prefilters each note's **whole body** (SIMD regex / `contains`; built lazily once a scan outlives 128 notes so short scans don't pay the compile) before the per-line pass — a rare/no-match query (the old worst case) is ~0.5ms @1k, ~3ms @10k, **~15ms @50k** (was ~95ms); matching queries are 15–20% faster; no DB or index to build or sync. `scan_notebook` reads files across scoped threads and path-sorts the records (deterministic `[[link]]` tie-breaks) — ~8ms @1k, ~89ms @10k, one stat per file. The **watcher** applies targeted upserts/removes for unambiguous create/modify/remove events and full-rescans for everything else (renames, mixed batches, any error). `[profile.release]` is `opt-level = 3` + fat LTO. On the JS side: **typing never re-renders React** (the session snapshot bails on no-change) and cursor motion doesn't rebuild editor decorations (skipped when the active-line set is unchanged); the **sidebar virtualizes** above 100 notes (`@tanstack/react-virtual`; the plain list renders verbatim below) and rows are memoized (an autosave re-renders 1 row); the **first paint** neither parses the editor chunk (lazy, §editor) nor waits on IPC (config + last-notebook load in parallel). **Backlinks** run as a Rust command (`commands::backlinks` → `links.rs`, hash-map resolution, mirrors `src/links.ts`) so the scan stays off the JS thread and only references cross IPC.

Lint/format are oxc, run from the repo root (not per-package, not through Turbo): `oxlint` (`.oxlintrc.json`, correctness=error/suspicious=warn, react+typescript+import plugins) and `oxfmt` (`.oxfmtrc.json`; pinned exactly — still beta). The codebase is kept oxfmt-formatted, **including `styles.css`** (so it's no longer byte-identical to the design source — diff semantically, not line-for-line). Tests: **Vitest** (`pnpm test`, node env — pure modules only, no jsdom/CM6) covers `autosave.ts` (incl. the cross-note regression), `settings.ts` round-trip, and the mock backend; **cargo test** (`pnpm test:rust`) covers `notebook`/`search`/`state`/`links`/`frecency` (`#[cfg(test)]` modules); **Playwright e2e** (`apps/desktop/e2e`, `pnpm e2e`) drives the web build + mock backend on :1420 in chromium AND webkit (≈ WebView2 / WKWebView parity) — it runs in CI, so UI changes (settings rows, picker markup, chords) must update the specs. `pnpm typecheck` + `pnpm lint` + all three test layers are the gates.

## Architecture: the parts that span files

**The editor is CodeMirror 6 + `@replit/codemirror-vim`** (`apps/desktop/src/editor/`), not a bespoke engine — the old `src/vim.ts` reducer is gone. Ex-commands, the `<Space>` leader palette, `gf`, and `:`-actions dispatch through a module-level handler registry: `ex-commands.ts` (`Vim.defineEx`/`Vim.map` + `setActiveHandlers`) → `app.tsx`. To follow any save/quit/open/follow flow, read `editor/editor.tsx` (mount + handler wiring) + `editor/ex-commands.ts` (where actions/ex-commands originate) + `app.tsx` (the handlers) together. `live-preview.ts`/`wikilinks.ts` add reveal-on-cursor-line decorations (viewport-limited, and skipped entirely on selection-only updates when the active-line set is unchanged — cursor motion must not rebuild them). **The editor chunk is lazy**: `app.tsx` has NO static value import from `editor/` — `Editor` is `React.lazy` over a module-scope `import()` (fetch kicks immediately, the ~700KB CM6+vim parse stays off the first-paint path), and vim prefs flow through the CM-free `editor/vim-config.ts` bridge (`setInsertEscape`/`setUserKeymaps` live there; `ex-commands.ts` registers the real Vim appliers when the chunk evaluates). Adding a value import from `app.tsx` (or anything eager) into `editor/*` silently re-eagers the chunk — check `dist/index.html` modulepreloads if you touch imports. `theme.ts` maps the CSS-var design tokens into the editor.

**`editor/commands.ts` is the single command table.** Every command (id, title, chord, ex-names, leader key, `editor`-action) lives here; the vim ex-commands, the `<Space>` which-key palette, the always-on `Mod-` chord keymap, the searchable command palette (`CommandSearch`), the `Mod-/` cheatsheet, and button tooltips all DERIVE from it — add or rebind a command in one place. `use-global-chords.ts` is the document-level chord fallback for the no-note-open state (gated on overlay React state, not `activeElement`). In-note find is `@codemirror/search`, toggled via `Mod-f`. Non-vim users are first-class; `bind <chord> <command-id>` lines in `~/.notesiderc` remap chords. The `Mod-/` cheatsheet (`components/cheatsheet.tsx`) is itself the **in-app keymap editor** — focus a row, `Enter` records a chord, `Del` unbinds, `r` resets, with clash detection via `chordConflict` (commands.ts); edits write `cfg.chords` (→ `bind` lines). The always-on chord keymap lives in a CM6 `Compartment` (`editor.tsx`) that App reconfigures on a `cfg.chords` change, so a rebind applies to the open editor with no remount (a remount would steal focus from the open cheatsheet).

**Do NOT wrap the desktop app in `<React.StrictMode>`.** `apps/desktop/src/main.tsx` deliberately omits it. StrictMode's dev double-invoke of effects would double-run the editor's mount effect (creating two `EditorView`s and firing the action-flush twice → double saves/quits/finder-opens). The landing app _does_ use StrictMode (no such pattern).

**The per-buffer state machine lives in `editing-session.ts`** — a framework-free `createEditingSession` store (working/saved/dirty, autosave, open/save/quit, config-as-a-buffer-kind, external reconcile, the editor remount `key`), consumed via `use-editing-session.ts` (`useSyncExternalStore`). It imports no React, so the whole loop is node-testable (`editing-session.test.ts`). `app.tsx` drives the session and owns the sidebar list, config _semantics_ (`onConfigApply`), and panels — it **no longer holds the working/saved maps**. The Editor is remounted via its `key` (`session.editorKey` + a vim suffix only — preview/relativeNumbers reconfigure live via compartments, see §config) so cursor position resets cleanly. **`editorKey` deliberately excludes `activeId`** (it keys on `gotoLine:reloadNonce:navSeq`, and `navSeq` bumps on every `open`) — so **rename-on-save** can migrate the buffer's id in place without remounting (no cursor jump). On an **explicit save** (`:w`/Mod-s, never autosave — avoids churn while the title is being typed), and only when the write **landed** (`persistNote` resolves with the saved meta — a failed save must not rename from stale disk content), the session **pre-checks the meta's title client-side** (`stemMatchesSlug`/`slugifyTitle` from links.ts — the same rules the backends apply) and skips the rename IPC when the filename already matches (the common `:w` costs one round-trip, not two); otherwise it calls `backend.renameNote(id)`; if the file's slug no longer matches its title the backend renames it **within the note's own directory** (nested notes are never hoisted to the root) and returns a new meta. The session then **re-pins any queued autosave** to the new id (`autosaver.repin` — a keystroke during the rename IPC would otherwise recreate the old file), migrates `lastNoteId`, migrates `activeId`/`noteTitle` **only if the buffer is still active** (a mid-rename switch must not clobber the new buffer's title), and fires `onNoteRenamed(oldId, meta)` (App swaps that sidebar row). Wikilinks resolve by title, so title-based `[[links]]` survive the move.

**The `~/.notesiderc` config buffer is a synthetic note.** `activeId === "config"` swaps the editor onto `configText` instead of a real note; `:w` runs `parseConfig` (settings.ts) and applies it. `settings.ts` round-trips `Config ⇄ text` via `serializeConfig`/`parseConfig`; the in-app `SettingsPanel` and the config buffer are two editors over the same `Config`. Config changes are applied by writing CSS custom properties (`--editor-font`, `--editor-size`, `--ui-scale`, the theme palette, etc.) onto `document.documentElement` in an effect in `app.tsx` — **split from persistence**: the var-writing effect keys on the visual fields only, and `backend.setConfig` is debounced ~300ms (flushed on `pagehide`/unmount) so held settings-steppers don't do an IPC/localStorage write per repeat. The applied theme is also mirrored to `localStorage["noteside:boot-theme"]` (id + mode + inline vars): the `index.html` head script paints it and `bootConfig()` seeds the initial `cfg` from it, so a dark/base16 user never sees a light first frame while the config store loads (the store remains the source of truth). Beyond the obvious fields, `Config` carries `theme` (a **theme id string** — see the Themes paragraph below; there is no separate accent field, themes own the full palette) and `uiScale` (a 90–130% **UI-chrome** scale — `--ui-scale` multiplies the chrome `font-size`s via `calc()`; the editor is immune since it sizes from `--editor-size`) and `relativeNumbers` (gutter, default **off** = absolute). **Only a vim-mode change remounts the editor**; `relativeNumbers` and `preview` reconfigure the live view through CM6 `Compartment`s (like the chord keymap) — a remount there would lose cursor + undo history and reseed from open-time text. `font-size` floors at 16.

**Search lives behind the `backend/` seam.** The `Backend` interface (`src/backend/types.ts`) exposes `searchFiles(query) → FileHit[]` and `searchContent(query, mode) → ContentHit[]`. The Tauri adapter (`backend/tauri.ts`) `invoke()`s the Rust commands `search_files`/`search_content` (`src-tauri/src/search.rs`: nucleo `fuzzy_files` for path + title matching (separate `positions`/`titlePositions` highlight ranges), a `content_search` scan for plain/regex/fuzzy), and the in-memory `mock.ts` backs browser dev + the landing demo. `Finder` only talks to the `Backend` interface, so it's identical in both modes — and search stays fully in-memory (no database). **Frecency ranks the finder** (`src-tauri/src/frecency.rs` + the same math in mock.ts): `EditingSession.open()` fire-and-forgets `backend.recordOpen(id)` on every USER navigation (never on watcher reloads), which bumps a 7-day-half-life decayed score persisted per notebook in `frecency.json` under the app data dir (pruned to 500). The empty finder query sorts recents by (pinned, effective frecency, updated) — never-opened notes keep the old order — and non-empty queries get a bounded ≤+15% score nudge so text relevance stays dominant; entries follow renames and drop on delete. An empty frecency map is output-identical to pre-frecency ranking (regression-pinned in the Rust tests).

**`links.ts` is the wikilink seam — pure, CodeMirror-free.** `parseWikilinks` / `resolveLink` (title → exact filename → filename-slug → title-slug, ordered so ties are deterministic and empty/punctuation targets never match) / `wikilinkAt` (link under a cursor, half-open end) / `urlAt` (the openable http(s)/mailto URL under a cursor — bare or a markdown `[text](url)` target) / `computeBacklinks`. The editor side (`editor/wikilinks.ts`) styles `[[links]]` and hides the `[[ ]]` syntax in live-preview using the _same_ reveal-on-cursor-line model as `live-preview.ts` (and exports `activeLines` from there), plus `[[`-autocomplete over note titles. `gf` / `gx` / `:follow` (exCommands → `App.onFollowLink` / `onOpenUrl`) is DWIM — a wikilink opens its note, otherwise the external URL opens in the system browser via `open-external.ts` → `tauri-plugin-opener` (`Mod`-click does the same; outside Tauri it falls back to `window.open`); the backlinks panel (`<Space>l` / `:backlinks`) calls the Rust `backend.backlinks(id)` command (the JS `computeBacklinks` — now map-based, O(notes + links) — backs only the mock/demo). Wikilinks are **not** lezer-markdown nodes — they're regex-scanned and never rewritten, so vim motions stay on the literal `[[Target]]` text. To move "jump to line" reliably, `openNote` bumps `navSeq` (part of the Editor remount `key`) on every nav.

**Themes are a token-application feature, not an editor feature (`themes.ts`).** `Config.theme` is a theme id. `themes.ts` holds the registry (`THEMES`) + the pure `schemeToPalette`/`applyThemeVars` mapping, all node-testable. Two kinds: **builtin** (`noteside-light`/`noteside-dark` — the app's own look; they apply **no** overrides so the `[data-theme]` blocks in `styles.css` render verbatim = the zero-regression default path) and **base16** (a curated set of ~31 MIT schemes vendored from tinted-theming into `src/bundled-schemes.json`). A base16 scheme's 16 slots map onto ~12 primitive CSS custom props written **inline on `<html>`** (inline > the `[data-theme]` selector), and the `color-mix`-derived tokens (`--sel`/`--active-line`/`--desk-a`/`--desk-b`) then recompute from those inline primitives — so the whole app **and the CM6 editor re-skin live with no remount** (theme is deliberately excluded from the editor remount `key`; `editor/theme.ts` is 100% `var(--x)`). `THEME_VARS` (the 11 props themes set) are cleared for builtin so the block wins. Each theme declares `mode` → drives `data-theme`. `resolveThemeId` accepts an id, a label, or the `light`/`dark` aliases (old configs keep working, zero migration). `--danger` is a real token now (extracted from inline reds); the dark `--desk-*` gradient derives from `--paper` (not hardcoded oklch) so cool themes re-skin the backdrop. The `:theme`/`:colorscheme` command + `<Space>t` open `components/theme-picker.tsx` (a **two-column dark·light** live-preview picker that opens on the _current_ theme: ↑↓ move within a column, ←→/Tab switch columns, applies on highlight, commits on Enter, reverts on Esc). **User-authored theme files are a deferred v2** — the plan is a Rust `~/.noteside/themes` scanner that parses base16 into the same `Base16Scheme` shape and reuses `schemeToPalette`, so nothing in `themes.ts` needs to change. base08–0F syntax-color theming for the editor is also out of scope for v1.

**Styling is a verbatim port of the design system, not Tailwind-in-the-app.** `apps/desktop/src/styles.css` is the design's hand-written CSS (oklch token palette, themed via `[data-theme="light"|"dark"]`). Class names (`av-*`, `set-*`, `fnd-*`) are load-bearing — the React components reproduce the prototype's exact markup against them. Treat `styles.css` as the visual source of truth; don't casually refactor it into utility classes. (This applies to the **desktop app only**. The **landing** has been migrated to **Tailwind v4 utilities**: `apps/landing/src/styles.css` now holds just the oklch tokens — exposed to utilities via `@theme inline` so `bg-paper`/`text-ink`/`font-serif`/`text-accent` follow `data-theme` — plus `@theme` keyframes and a thin `@layer components` for the reusable/bespoke bits (`.wrap`, `.btn*`, the logo `.bmark`/`.wordtype`, keycast `.kc-*`, `.reveal`). Everything else is utilities in `app.tsx`. `--breakpoint-sm` is overridden to `45rem` to match the design's single 720px breakpoint, used via `max-sm:`.) Cross-cutting bits in the desktop `styles.css`: a `:focus-visible` accent ring (keyboard-only), a `prefers-reduced-motion` block, custom thin scrollbars (`scrollbar-width` + `::-webkit-scrollbar`, `--ink-faint`-derived thumb), and a trailing **legibility** layer that floors the smallest chrome font-sizes — every chrome `font-size` is `calc(Npx * var(--ui-scale))`.

**Window chrome is native, with a macOS overlay titlebar.** The Tauri window is `decorations: true` (see `src-tauri/tauri.conf.json`); on **macOS** it uses `titleBarStyle: "Overlay"` + `hiddenTitle: true` + `trafficLightPosition`, so the OS draws the traffic lights over the app's toolbar (one unified bar) and the app insets that toolbar's left edge (`html.tauri.macos .av-titlebar { padding-left }`, macOS flagged via a `macos` html class in `main.tsx`). Windows/Linux get a plain native title bar above the toolbar (a follow-up could add custom caption buttons there for the same unified look). The in-app decorative traffic lights (`TrafficLights`) and the custom titlebar title are **demo-only**: gated out of the native app in JSX (`!isTauri()` in `app.tsx`), rendering only in the **landing demo** (`?embed=1` iframe, no native frame), where red closes the note buffer and amber/green are inert. `isTauri()` / the `embed` + `tauri` html classes gate all of this; the same build runs both as the native app and the embedded demo (which fills its iframe edge-to-edge via the `embed` rules).

**Landing ↔ desktop coupling is the demo iframe only.** The landing embeds the desktop web build via `<iframe src={DEMO_URL}>`; in dev `DEMO_URL` points at the desktop Vite server, in prod at `/demo/` (populated by `pnpm demo:build`). Override with `VITE_DEMO_URL`. Deploys on Vercel via `vercel.json` (repo root): build = `pnpm build:landing` (builds the demo into `public/demo`, then the landing), output `apps/landing/dist`, **Root Directory = repo root** (a default `vite build` would ship an empty `/demo`).

**SEO + domains.** Landing → **noteside.app**, docs → **docs.noteside.app** (a _separate_ Vercel project, Root Directory = `apps/docs`, config `apps/docs/vercel.json` with the SPA-fallback rewrite). Landing SEO is static in `apps/landing/index.html` (canonical/OG/Twitter/theme-color + WebSite/SoftwareApplication JSON-LD) plus `public/robots.txt` + `public/sitemap.xml`. Docs SEO is per-page: `routes/docs.tsx`'s clientLoader component renders `<title>`/`<meta>`/canonical/OG/Twitter/JSON-LD in the body and **React 19 hoists them into the prerendered `<head>`** — do NOT add a route `meta()` (React 19 hoists but does NOT dedupe → duplicate tags). The docs sitemap is a prerendered resource route (`routes/sitemap.ts`, registered in `routes.ts` and pushed into `react-router.config.ts` `prerender()`); robots is static `public/robots.txt`. Canonical + sitemap URLs are clean directory paths with **no trailing slash** (root = `https://docs.noteside.app`). The **OG cards** (`apps/{landing,docs}/public/og.png`, 1200×630, opaque) render from `apps/desktop/scripts/og.html` via headless Chrome — regenerate with `pnpm og:build` (must be online; one template, `?variant=landing|docs`, `--force-device-scale-factor=1`). ## Releases

**Releases are deliberate and batched — a manual GitHub Action, not per-push.** Run the **Release** workflow (`.github/workflows/release.yml`, `workflow_dispatch`; set the `dry_run` input to preview the version/notes) when you want to ship the desktop app; the landing/docs keep their own continuous Vercel cadence, so the two are decoupled. The pipeline: **semantic-release** (`.releaserc.json`, conventionalcommits preset) computes the version, writes `CHANGELOG.md`, bumps the version files, tags, and creates the GitHub Release; then a 4-target `tauri-action` matrix (macOS arm64/x64, `ubuntu-22.04` + `libwebkit2gtk-4.1-dev`…, windows) attaches binaries to it. The CI gate (`.github/workflows/ci.yml`) runs typecheck/lint/format/test + `cargo check`/`test` on PRs and pushes to main.

The version is **derived from conventional commits** (`feat`→minor, `fix`/`perf`/`polish`/`refactor`→patch, `feat!`/`BREAKING CHANGE`→major), so commit hygiene is load-bearing. `node scripts/bump-version.mjs <version>` is the single sync point — it regex-patches the `version` in **8 files**: `apps/desktop/src-tauri/tauri.conf.json` (the binary's source of truth), `Cargo.toml`, the `noteside` entry in `Cargo.lock`, and the root + 4 app `package.json`s. Those `0.1.0`s are **placeholders semantic-release overwrites at release time — don't hand-bump them**; the first release auto-cuts `v1.0.0` (no prior tags). Keep evergreen prose version-agnostic and point download CTAs at `…/releases/latest`. Builds are currently **unsigned** (one-time Gatekeeper/SmartScreen prompt) with no auto-updater. Setup gotcha: the repo's _Settings → Actions → Workflow permissions_ must be **read+write**, or semantic-release can't push the tag/commit/release.

## Conventions & gotchas

- **TypeScript is strict** with `noUnusedLocals`, `noUnusedParameters`, and `verbatimModuleSyntax` (from `tsconfig.base.json`). Use `import type { … }` for type-only imports or the build breaks.
- **Source files are kebab-case** (`settings-panel.tsx`, `editing-session.ts`, `use-global-chords.ts`, `ex-commands.ts`) — never CamelCase. Exported component/identifier names stay PascalCase (`export function SettingsPanel`); only the _filename_ is kebab. `apps/docs` keeps React Router's own lowercase route-file convention. The macOS filesystem is case-insensitive, so a stale wrong-case import resolves locally but breaks the Linux CI build — rename with a two-step `git mv` and fix every import path.
- **The mock backend is an in-memory seed** (`src/data.ts`, used in browser dev / the landing demo); the real Tauri backend persists to Markdown files. Notebook scanning, atomic writes, and the watcher live in `src-tauri/src` (`notebook.rs`/`commands.rs`/`watcher.rs`); search stays fully in-memory (no database). The notebook index is **`Arc<Vec<Arc<NoteRecord>>>`** (`state.rs`): searches snapshot the outer Arc and run outside the lock; a mutation racing a live snapshot copies N pointers and rebuilds only the touched record — never the note bodies (inner records are immutable values: build a new `Arc<NoteRecord>`, don't mutate one). The vector is **path-sorted everywhere** (scan sorts, `upsert_sorted` inserts at `partition_point`) and every by-path lookup goes through `state::find_record` (binary search) — keep both invariants together or lookups silently miss. The `rename_note` command (rename-on-save, §editing-session) slugs the title, skips a no-op via `stem_matches_slug`, `fs::rename`s to a `unique_note_path`, and calls `state::record_own_rename` — which drops the old record, upserts the new, and arms the same `suppress_until` window `record_own_write`/`record_own_delete` use, so the watcher ignores the move (a rename surfaces as delete+create) instead of clobbering the live buffer. For external events the watcher first tries a **targeted update** (`watcher.rs::classify_events`/`targeted_updates` → `state::apply_external`): unambiguous create/modify re-reads just those `.md` files, remove retain-drops them; rename kinds, mixed/multi-path batches, non-.md paths, or any read error fall back to the full rescan — when in doubt, rescan.
- **The desktop app bundles fonts locally** (`apps/desktop/src/fonts.ts`) so it runs **fully offline** — no Google Fonts request on launch. It does NOT import the `@fontsource/*` CSS (which ships every face in woff2 **and** woff — ~1.75MB of assets): fonts.ts imports the **woff2 files directly** (all subsets each family ships — latin/latin-ext plus cyrillic/greek/vietnamese where they exist; 74 faces across Newsreader/Spectral serif, IBM Plex/JetBrains/Space mono) and injects self-authored `@font-face` rules (family/weight/style/`unicode-range` copied verbatim from the fontsource CSS) at module eval — ~868KB total, zero `.woff`, glyph coverage identical to @fontsource. Adding a weight/style = add its subset woff2 imports + `face()` lines there. The **landing + docs** are websites, so they load Newsreader + IBM Plex Mono from Google Fonts (`index.html` / `root.tsx`); the brand guide too. CSS stacks fall back to system serif/mono everywhere.
- **Tauri Rust is a single crate** at `apps/desktop/src-tauri` (no root cargo workspace). Turbo does not cache the Rust build — `desktop#build` is the _web_ bundle; `tauri build` is separate. Window-control permissions and `data-tauri-drag-region` are authorized in `src-tauri/capabilities/default.json` (window label `main`).
- **Toolchain: Node 24 + pnpm 11** (`.nvmrc`, `packageManager` in root `package.json`). pnpm 11 enforces a **supply-chain freshness gate** (`minimumReleaseAge: 1440` in `pnpm-workspace.yaml`) — if `pnpm install`/`pnpm <script>` fails with `ERR_PNPM_MINIMUM_RELEASE_AGE_VIOLATION`, a dependency was published <1 day ago. Wait for it to age, or allowlist that one package under `minimumReleaseAgeExclude` (never `minimum-release-age=0` globally). Dependency build scripts are also gated — approve trusted ones under `allowBuilds` (e.g. `esbuild`). When the gate blocks a `pnpm <script>` (pnpm 11 re-runs an install check first), run the tool binary directly to bypass it: `./apps/desktop/node_modules/.bin/vitest run --root apps/desktop` (vitest is package-local — not in the root `.bin`), `./node_modules/.bin/tsc --noEmit -p apps/desktop/tsconfig.json`, `./node_modules/.bin/oxlint apps/desktop/src`, `./node_modules/.bin/oxfmt --check apps/desktop/src`, `(cd apps/desktop && ./node_modules/.bin/tauri icon …)`, `cargo test --manifest-path apps/desktop/src-tauri/Cargo.toml`.
- **Editor theme/extension changes need a full reload, not HMR.** CM6 injects `EditorView.theme(...)` CSS as a static StyleModule and wires extensions at `EditorView` mount, so Vite HMR won't apply `editor/theme.ts` or extension edits to a running editor — restart/hard-reload to see them. Editor selection/cursor colors must use `!important` + CSS vars: CM's baseTheme defaults win on specificity and the theme isn't declared `dark`, so a plain `.cm-selectionBackground` rule loses (CM's light default `#d7d4f0` then shows in **both** themes).
